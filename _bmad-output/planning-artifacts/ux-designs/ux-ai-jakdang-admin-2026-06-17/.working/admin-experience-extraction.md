## 1. Foundation (기반)

### 1-1. Form-factor (폼팩터)

- **데스크톱 우선(desktop-first) / 사실상 데스크톱 전용** `[ASSUMPTION]`
  - 근거(직접): 광고 등록 항목에 `PC / 모바일 구분`, 광고 성과 지표에 `PC / 모바일별 성과`가 있는데, 이는 **관리 대상 데이터(광고 노출)** 의 구분일 뿐 **관리자 화면 자체의 반응형 요구**는 아니다. 따라서 관리자 콘솔 화면의 모바일 대응은 소스에 명시되지 않음.
  - 근거(구조): 관리자 앱은 사이드바/상단바를 갖는 `AdminShell` 레이아웃(`apps/admin/components/layout/AdminShell.tsx`), 테이블·필터·드로어·모달 중심의 백오피스 UI 시스템(`packages/admin-design-system`)을 사용 → 전형적 데스크톱 관리 콘솔 패턴.
  - 결론: 관리자 폼팩터는 **데스크톱 브라우저 콘솔**로 간주. 모바일 최적화는 비요구로 `[ASSUMPTION]`.
- 별도 서브도메인(`admin` 서브도메인 등)으로 분리 배포, 사용자 사이트(포트 3003)·API(4003)와 다른 포트(3004)로 독립 실행.

### 1-2. UI System (명명된 UI 시스템)

- **관리자 전용 디자인 시스템 = `@ai-jakdang/admin-design-system`** (프레임워크 비종속, 순수 CSS + 바닐라 JS 패키지, 위치 `packages/admin-design-system/`).
- 사용자 사이트와 **완전 분리**: 디자인 토큰·색상·타이포·간격·라운드·그림자·CSS·UI 컴포넌트·레이아웃·반응형 규칙·아이콘·버튼/입력/셀렉트/모달 디자인을 일체 공유하지 않음. 관리자 앱은 사용자 사이트의 `styles/* components/* features/*`를 import 하지 않는다.
- 관리자 앱이 공유받는 것은 **비시각적 패키지만**: `@ai-jakdang/contracts`(API 요청/응답 타입·Zod 스키마), `@ai-jakdang/auth`(권한 타입, `canAccessAdmin` 등 권한 검사 규칙).
- 토큰 네임스페이스: `--admin-*` (중립 토큰). 브랜드 교체 지점은 `css/tokens/theme.css`(Primary 색상·브랜드) + 로고.
- 제공 컴포넌트(시각): 버튼·카드·폼·**테이블**·**배지**·**알림**·**모달**·**드로어**·**토스트**·차트.
- 제공 인터랙션(JS): 사이드바·**셀렉트**·**오버레이**·**토스트**·**탭**·**태그입력**·**테이블**·차트. 진입점 `initAdminUI()`.
- 아이콘: `remixicon`.
- 현재 상태: 디자인 시스템 패키지는 구축·연동 완료(빌드 통과). 단, 로그인/대시보드/`AdminShell`은 아직 **자리표시(placeholder)** 상태이며 디자인 시스템 컴포넌트로 교체 예정. 인증/권한 연동, 실제 데이터 연동, 권한별 메뉴는 **미구현**.

## 2. Information Architecture (정보구조) — Admin IA Tree

관리자 메뉴 최종 구조(소스 §2)는 13개 1차 메뉴. 인증 라우트(`/login`)와 진입 리다이렉트(`/` → `/dashboard`) 포함.

```text
관리자 콘솔 (admin 서브도메인 / :3004)
├── /login                         로그인 (인증 단계 구현 예정, 비로그인 진입점)
├── / → /dashboard                 루트 진입 시 대시보드로 리다이렉트
│
├── 1. 대시보드 (Dashboard)        [1차·핵심]   /dashboard
│   ├── 상단 요약 카드 (방문자/페이지뷰/신규회원/글/질문/다운로드/신고)
│   ├── 운영 알림 (답변대기 질문 / 미처리 신고 / 신고 많은 자료)
│   ├── 콘텐츠 성과 (인기 글 / 인기 질문 / 인기 자료)
│   └── 마케팅 성과 (검색 유입 상위 / 가입 전환 상위 / 다운로드 전환 상위)

├── 2. 접속 통계 (Analytics)       [1차·핵심]

│   ├── 유입 경로 분석 (검색/직접/SNS/외부링크/광고/AI검색추정/기타)

│   │                 외주·판매 팁, 수익화 사례, AI 창작마당, 내가 만든 AI 제품
│   ├── 목록(검색/게시판·작성자·상태·공지·추천·메인노출·기간 필터)
│   └── 행 처리(보기/수정/공지/고정/추천/메인노출/숨김/삭제/복구)

├── 4. 묻고답하기 관리 (Q&A)       [1차·핵심]
│   ├── 목록 필터(전체/답변대기/답변있음/해결됨/신고있음/도움된답변있음)
│   ├── 질문 처리(보기/수정/숨김/삭제/복구/상태 강제 변경)
│   └── 답변 처리(보기/숨김/삭제)
│
├── 5. 실전자료 관리 (Resources)   [1차·핵심]
│   ├── 목록 필터(유형/지원환경/난이도/상태/신고/작성자/등록일/다운로드/평점)

├── 6. 댓글·후기 관리 (Comments)   [1차·핵심]

│
├── 7. 신고 관리 (Reports)         [1차·핵심] ★통합 화면

│   ├── 처리 상태(접수/확인중/처리완료/반려)
│   └── 처리(대상보기/사유·신고자·작성자 확인/숨김/삭제/반려/이용제한/메모/상태변경)
```

### 2-1. Navigation (내비게이션)

- **좌측 사이드바 + 상단바** 셸(`AdminShell`)에 13개 1차 메뉴 배치 `[ASSUMPTION]` (디자인 시스템에 사이드바 인터랙션 존재; 메뉴 트리는 위 13개 구조 직접 매핑).
- 진입: 로그인 → `/dashboard`. 루트(`/`)는 항상 대시보드로 리다이렉트.
- **권한별 메뉴 노출은 미구현**이며 향후 단계 과제(문서 §6, "권한별 메뉴").
- 크로스 링크: 댓글·후기 관리 → "관련 게시글로 이동", 신고 관리 → "신고 대상 보기"(대상 콘텐츠로 이동) 등 화면 간 이동 동선 존재.

### 2-2. 개발 우선순위(IA 깊이)

- **1차·핵심**: 대시보드, 접속 통계, 게시글, 묻고답하기, 실전자료, 댓글·후기, 신고, 회원, 광고.
- **1차·기본형**(얕게 시작): 포인트, 등급, 뱃지.
- **후순위/제외**: 포인트 교환소, 유료 자료 정산, 결제 관리, 뱃지 미션, 복잡 랭킹, 자동 파일 보안검사, AI 자동검수, 실시간 알림, 쪽지/채팅 관리, 운영자 활동 로그 상세 분석, A/B 테스트·마케팅 캠페인 자동화.

## 3. Operator Roles / Permissions (운영자 역할·권한)

소스 §11-3, §17에 명시된 3단계 권한 모델.

| 권한 | 관리자 접근 | 핵심 가능 범위 | 제한 |
|---|---|---|---|
| **최고관리자** (super admin) | 전체 | 모든 관리자 기능, 운영자 권한 부여, **사이트 설정 변경**, **광고 관리**, 회원 권한 변경, **삭제 처리** | 없음 |
| **운영자** (staff/moderator) | 일부 | 게시글·댓글·신고·실전자료·묻고답하기 관리, 회원 **일부 조회** | 사이트 핵심 설정·광고 수익 설정 수정 불가(권장), 회원 권한 변경 불가 `[ASSUMPTION]` |
| **일반회원** | 불가 | 관리자 페이지 접근 불가 | 전면 차단 |

- 권한 검사 규칙은 비시각 패키지 `@ai-jakdang/auth`(`canAccessAdmin` 등)에 위치.
- 주의: "운영자는 핵심 설정/광고 수정 **못 하게 하는 것이 좋다**"는 **권고 표현**이며 강제 enforcement 여부는 미확정 `[ASSUMPTION]`.
- "운영자가 삭제 처리 가능 여부": §17-1은 삭제 처리를 최고관리자 능력으로 명시하나, §6/§7 등 운영자 담당 메뉴에도 "삭제" 기능이 존재 → **운영자의 삭제 가능 범위가 모순/불명확** (Open Question 참고).

## 4. Voice and Tone (보이스/톤 — 운영자 마이크로카피)

- 소스에 **명시적 UI 마이크로카피(버튼 문구·확인 다이얼로그 문구·토스트 문구)는 없음** — 기능명/상태명 수준만 존재.
- 추출 가능한 **도메인 어휘(운영 라벨)**:
  - 상태 라벨: `공개 / 숨김 / 삭제`, 회원 `정상 / 이용 제한 / 탈퇴`, 신고 `접수 / 확인중 / 처리완료 / 반려`, Q&A `답변대기 / 답변있음 / 해결됨`.
  - 행동 동사: `숨김 처리 / 삭제 처리 / 복구 처리 / 반려 / 이용 제한 / 메모 작성 / 강제 변경 / 회수 / 수동 지급·차감`.
  - 브랜드 톤 등급명: `새내기 / 작당원 / 실전러 / 고수 / 마스터 / 운영자` (브랜드 톤에 맞게 수정 가능 명시).
- 톤 방향성(기획 의도): 대시보드는 "예쁜 차트가 아니라 **오늘 무엇을 봐야 하는지** 알려주는 운영 중심" → **실무적·간결·결정 유도형** 톤 `[ASSUMPTION]`.
- 신고 처리 원칙의 신중함("신고 들어와도 무조건 삭제하지 않는다") → 파괴적 액션 카피는 **판단을 환기하는 톤** 권장 `[ASSUMPTION]`.

## 5. Component Patterns (행동 기반 컴포넌트 패턴)

각 관리 화면이 공통적으로 따르는 **목록(테이블) + 필터 + 행 액션** 골격.

### 5-1. 테이블(목록)
- 거의 모든 1차 메뉴가 **컬럼형 목록 테이블**을 가짐(게시글/Q&A/실전자료/댓글/신고/회원).
- 표준 컬럼군: 제목·작성자·작성일 + 활동지표(조회/댓글/좋아요/다운로드/신고 수) + **상태** + 운영 플래그(공지/고정/추천/메인노출).
- 디자인 시스템 `table` 컴포넌트 + JS 테이블 인터랙션 제공.

### 5-2. 필터 바
- 메뉴별 전용 필터셋: 게시판별/상태(공개·숨김·삭제)/공지·추천·메인노출/기간/작성자/신고여부/자료유형·환경·난이도/Q&A 상태/댓글 유형 등.
- 셀렉트·태그입력·탭 인터랙션이 필터 UI 후보(디자인 시스템 제공).

### 5-3. 행 액션 / 벌크 액션
- 행 단위 처리: 보기·수정·숨김·삭제·복구·플래그 토글(공지/고정/추천/메인노출 설정·해제).
- **벌크 선택(다중 선택 후 일괄 처리)**: 소스에 명시 안 됨 → 백오피스 통상 패턴으로 필요 추정 `[ASSUMPTION]`. (Open Question)

### 5-4. 폼 / 모달 / 드로어
- 설정·수정·규칙편집 화면(사이트 설정/광고 등록/포인트·등급·뱃지 규칙)은 **폼** 중심.
- 디자인 시스템에 **모달·드로어·오버레이·토스트** 제공 → 상세보기/수정/확인은 모달 또는 드로어, 처리 결과는 토스트로 노출 `[ASSUMPTION]`.
- 신고/회원 상세는 "메모 작성", "처리 상태 변경" 등 인라인 편집형 필드 포함.

### 5-5. 대시보드 위젯
- 요약 카드(KPI), 알림 리스트(운영 알림), 랭킹 리스트(인기/전환 상위), 차트(접속 통계) — 카드 + 차트 + 리스트 조합.

## 6. State Patterns (상태 패턴)

소스가 직접 정의한 상태와, 화면 동작상 필요한 상태(추론) 구분.

- **데이터 상태(도메인)** — 직접 명시:
  - 콘텐츠/댓글: `공개 / 숨김 / 삭제`(+ 삭제 후 **복구** 가능 → soft-delete 패턴 시사).
  - 회원: `정상 / 이용 제한 / 탈퇴`.
  - 신고: `접수 / 확인중 / 처리완료 / 반려`.
  - Q&A: `답변대기 / 답변있음 / 해결됨`.
- **빈 상태(empty)**: 답변대기 0건, 미처리 신고 0건, 최근 자료 없음 등 대시보드 알림/리스트의 빈 상태 필요 `[ASSUMPTION]`.
- **로딩(loading)**: 테이블/차트/통계 로딩 상태 필요 `[ASSUMPTION]`.
- **저장중(saving)**: 규칙 설정·권한 변경·포인트 지급 등 쓰기 작업의 저장중/완료 토스트 `[ASSUMPTION]`.
- **에러(error)**: API 오류 표시 패턴 — 소스 미정의 `[ASSUMPTION]`. (Open Question)
- **권한 거부(permission-denied)**: 일반회원 접근 불가 + 운영자의 설정/광고 영역 접근 제한 → 권한 거부/메뉴 숨김 상태 필요. enforcement 방식 미정 `[ASSUMPTION]`.
- **자동 숨김(auto-hide)**: 신고 누적 시 자동 숨김 기준은 "초기에는 신중하게" → 기본 OFF 또는 토글, 오탐(정상 글 숨김) 리스크 인지됨.

## 7. Interaction Primitives (인터랙션 프리미티브)

| 프리미티브 | 소스 근거 | 비고 |
|---|---|---|
| 검색(search) | 게시글/작성자/회원/자료 검색 | 직접 명시 |
| 필터(filter) | 모든 목록 메뉴 전용 필터셋 + 통계 기간/유입 필터 | 직접 명시 |
| 정렬(sort) | 다운로드 많은/조회수 높은/댓글 많은 등 랭킹 | 정렬 컨트롤은 `[ASSUMPTION]` |
| 페이지네이션(paginate) | 대량 목록 전제 | 소스 미명시 `[ASSUMPTION]` |
| 기간 선택(date range) | 통계 기간 필터, 사용자 지정 기간, 광고 노출 기간 | 직접 명시 |
| 벌크 선택(bulk-select) | 일괄 처리 | 소스 미명시 `[ASSUMPTION]` |
| 인라인 편집(inline-edit) | 상태 강제 변경, 관리자 메모, 플래그 토글 | 부분 명시 |
| 토글(flag toggle) | 공지/고정/추천/메인노출 설정·해제, 자동지급·자동등급 ON/OFF | 직접 명시 |
| **파괴적 확인(confirm-destructive)** | 삭제/이용제한/회수/차감 | 확인 UX 자체는 미명시 (§7-5 처리 원칙만 존재) `[ASSUMPTION]` |

### 파괴적/모더레이션 액션 + 확인 동작
- 파괴적 액션 목록: **삭제**(게시글/질문/답변/댓글/자료/첨부), **숨김**, **이용 제한(밴)**, **뱃지 회수**, **포인트 차감**, 신고 **반려**, **권한 변경**.
- **복구(restore) 존재** → 삭제·숨김은 되돌릴 수 있는 soft 처리로 설계됨(영구 삭제 여부 미정 `[ASSUMPTION]`).
- 확인 동작: 소스는 명시적 confirm 다이얼로그 문구/2단계 확인을 정의하지 않음. 단 §7-5 신고 처리 5단계(대상확인 → 사유확인 → 위반판단 → 숨김/삭제/반려 → 제재)는 **삭제 전 검토를 강제하는 절차적 가드레일**로, 모더레이션 액션에 **확인 단계가 필요함을 시사**. 구체 UX는 Open Question.
- 운영자가 직접 **하지 않는** 행위(가드레일):
  - Q&A: 질문자의 "도움된 답변" 대신 지정 금지(질문자 권한 유지).
  - 댓글: 회원 댓글 **내용 직접 수정 금지**(숨김/삭제 중심) — 신뢰 문제.
  - 실전자료: 검수됨/공식/운영자 인증/안전성 보증/자동 보안검사 **미제공**(책임 회피 의도).

## 8. Accessibility Floor (접근성 최소선)

- 소스에 **접근성 명시 요구 없음** `[ASSUMPTION]`.
- 기반 권장(데스크톱 콘솔 최소선) `[ASSUMPTION]`:
  - 키보드 내비게이션(테이블 행/필터/모달 포커스 트랩).
  - 상태/플래그를 색상만으로 구분하지 않기(배지에 텍스트 라벨 병기 — 디자인 시스템에 `badge`/`alert` 존재).
  - 파괴적 액션의 명확한 라벨·확인 + 포커스 이동(스크린리더 대비).
  - 차트는 표/수치 대체 제공.
- (Open Question: 접근성 목표 등급(WCAG) 미정.)

## 9. Key Flows (핵심 운영자 여정)

### Flow A — 신고 처리 (climax: "삭제할까, 살릴까")
- 주인공: **운영자 "지훈"**(staff). 매일 아침 대시보드부터 연다.
- 동선: 대시보드 `운영 알림 - 미처리 신고 N건` 확인 → **신고 관리**(통합 목록) 진입 → 신고 행 클릭 → 신고 대상 콘텐츠 보기 + 신고 사유·신고자·작성자 확인 → **실제 위반 여부 판단** → 처리 분기.
- **Climax beat**: 신고가 들어왔다고 무조건 지우지 않는다. 지훈은 대상 콘텐츠를 직접 열어 위반인지 가린다 — "정상 글을 잘못 지우면 신뢰가 깨진다". 위반이면 **숨김 또는 삭제**, 아니면 **반려**, 악질이면 작성자 **이용 제한** + 관리자 **메모** → 처리 상태를 `처리완료/반려`로 변경.
- 권한 경계: 운영자 가능 영역. 단 운영자의 삭제/제재 범위는 §17과 모순(Open Question).

### Flow B — 좋은 콘텐츠 메인 노출 (climax: "이 자료를 메인에 올린다")
- 주인공: **최고관리자 "은영"**.
- 동선: **접속 통계**에서 다운로드 전환율 높은 자료 발견(예: 조회 300/다운 80 = 26.6%) → **게시글/실전자료 관리**로 이동 → 해당 자료 행에서 **메인 노출 설정** + 추천 지정 + (필요 시)상단 고정.
- **Climax beat**: "조회수 1,000에 다운 20(2%)보다, 조회 300에 다운 80인 자료가 더 강한 자료다" — 은영은 숫자로 강한 자료를 골라 메인에 띄운다. 운영=마케팅 판단 도구라는 기획 철학이 작동하는 순간.

### Flow C — 위험 첨부파일 사후 관리 (climax: "악성 의심 파일을 내린다")
- 주인공: 운영자 "지훈".
- 동선: 신고(`악성 파일 의심`) 또는 실전자료 신고 내역 → **실전자료 관리** 상세 → **첨부파일 확인** → 부적절 첨부파일 삭제 또는 자료 숨김/삭제.
- **Climax beat**: AI작당은 "누구나 등록" 정책 + "운영자는 검수자가 아니라 사후관리자"라는 선을 지킨다. 지훈은 안전성을 **보증하지 않되**, 위험 자료는 빠르게 내린다. (자동 보안검사 없음 = 사람이 판단.)

### Flow D — 회원 활성화 보상 (climax: "수동으로 포인트/뱃지를 준다")
- 주인공: 최고관리자/운영자.
- 동선: **회원 관리** 상세 → 활동 내역(글/답변/도움된답변/자료) 확인 → **포인트 수동 지급** 또는 **뱃지 수동 지급** / 필요 시 **등급 수동 변경**.
- **Climax beat**: 자동 규칙으로 못 잡는 "기여한 회원"을 운영자가 직접 알아보고 보상해 참여를 끌어올린다. (단, 포인트는 현금성 보상과 비연결 — 활동 지표일 뿐.)

### Flow E — 위반 회원 제재 (climax: "이용 제한을 건다")
- 주인공: 최고관리자 "은영".
- 동선: 신고 누적/심각 위반 확인 → **회원 관리** 상세 → 신고 이력 보기 → **회원 상태 변경: 이용 제한** (+ 포인트 차감, 위반 콘텐츠 삭제).
- **Climax beat**: "신고 누적 후 위반 확정" 시점에만 제재 — 절차를 거친 제재. 밴은 되돌릴 수 있는 상태 변경(정상/이용제한/탈퇴)으로 관리.

### Flow F — 광고 수익 위치 운영 (climax: "신뢰를 해치지 않는 선에서 단다")
- 주인공: 최고관리자(운영자 접근 제한 영역).
- 동선: **광고 관리** → 광고 등록(유형/위치/PC·모바일/노출기간/코드) → 성과(노출/클릭/CTR) 모니터링.
- **Climax beat**: "다운로드 버튼 근처에 광고를 과하게 넣으면 신뢰가 떨어진다" — 수익과 신뢰 사이에서 위치를 절제. 광고 영역은 **최고관리자 전용** 권한 경계.

## 10. Open Questions (미해결 질문 — 결정 필요)

1. **운영자 삭제 권한 모순**: §17-1은 "삭제 처리=최고관리자"로 적었으나 게시글/Q&A/실전자료/댓글 관리(운영자 담당)에 삭제 기능이 있음. 운영자가 삭제 가능한가, 아니면 숨김까지만이고 영구삭제는 최고관리자인가?
2. **파괴적 액션 확인 UX 미정의**: 삭제/밴/회수/차감에 대한 confirm 다이얼로그(1단계 vs 2단계, 사유 입력 강제 여부, undo/복구 노출 방식)가 소스에 없음.
3. **벌크 선택/일괄 처리 여부**: 대량 신고·게시글 일괄 숨김/삭제가 필요한가? (소스 미명시)
4. **페이지네이션/정렬 컨트롤**: 목록 정렬 가능 컬럼, 페이지네이션 vs 무한스크롤 미정.
5. **삭제의 영구성**: 복구가 가능하므로 soft-delete로 보이나, 영구 삭제(hard-delete) 경로/보존기간이 있는가?
6. **권한 enforcement 강도**: 운영자의 설정/광고 제한이 "권장"인지 "강제 차단"인지. 권한별 메뉴 숨김 vs 접근 시 거부.
7. **회원 "일부 조회"의 범위**: 운영자가 볼 수 있는 회원 정보 범위(이메일 등 PII 포함 여부) 불명확.
8. **관리자 폼팩터 확정**: 데스크톱 전용인지, 태블릿/모바일 최소 대응이 필요한지 명시 필요.
9. **저니 없는 서피스**: 접속 통계 심화(키워드/유입 분석), 사이트 설정(SEO·파일정책·신고설정), 등급/뱃지 규칙 편집, 대시보드 마케팅 위젯은 화면 정의만 있고 **운영자 여정(누가 언제 왜)** 이 없음 — UX 흐름 정의 필요.
10. **접근성 목표 등급** 미정(WCAG 준수 수준).
