# AI작당 관리자(ADMIN) 사이트 — 제품 컨텍스트

## 1. 관리자 사이트의 목적 & 운영 주체

### 1-1. 운영 목적 (operational purpose)
관리자 사이트는 **커뮤니티의 사후(事後) 운영·관리 도구**다. AI작당은 비회원 읽기 개방 + 회원 즉시 등록(검수 없음) 구조이므로, 품질·안전·SEO 평판은 **사전 차단이 아니라 사후 모더레이션**으로 지킨다. 관리자 사이트가 그 사후 운영의 중심이다.

핵심 운영 목적:
- **콘텐츠 사후 관리**: 공지·고정·추천·메인 노출·숨김·삭제 (운영자만 가능, 회원 글쓰기 화면에는 미노출 — FR-10.1)
- **모더레이션**: 신고 큐 처리, 신고 누적 자동 숨김 검토, 회원 제재(경고/일시정지/영구정지) (FR-8.x, FR-10.2)
- **실전자료 사후 관리**: 자료 숨김/삭제, 부적절 첨부 삭제, 후기 댓글 관리 (FR-10.3)
- **SEO 보정**: 시스템 자동 생성된 메타(title/description)를 운영자가 보정 (FR-10.4, FR-11.1) — SEO가 본 제품 #1 과제이므로 운영 비중 높음
- **통계/대시보드**: 검색 유입·다운로드 추이 등 운영 지표 모니터링 (FR-10.5)
- **광고 관리**: 애드센스 등 광고 영역 운영 (FR-10.5, `[ASSUMPTION]` 상세 미확정)

### 1-2. 운영 주체 (roles)
- **운영자/관리자**: 사실상 **단일 권한군**. PRD 역할표(섹션 3)는 비회원/일반회원/운영자 3단계만 정의 — 관리자 내부 세분화(슈퍼관리자/모더레이터 등)는 **명시되지 않음**.
- 초기에는 **1인 운영(소수 운영자)** 전제. 기술 스택 문서가 "1인 개발 단계"를 반복 언급하므로 운영도 동일 규모로 추정. `[ASSUMPTION]`
- 운영자는 콘텐츠 **생산자**이기도 함 — 초기 핵심 글 50~100개를 직접 작성(메인 노출/상단 고정 지정 포함). 즉 관리자 UX는 "감시·처리" + "직접 발행/큐레이션" 두 모드를 함께 담아야 함.
- 기술 스택 문서 §9에 **"권한별 메뉴 접근(권한별 메뉴 노출)"** 항목이 자체 개발 범위로 명시됨 → 향후 다중 권한 등급 대비 설계는 필요. `[ASSUMPTION]` 현재 등급 정의는 없음.

## 2. 관리자 관련 기능 요구사항 (FR/NFR 인용)

### 2-1. 콘텐츠/게시물 관리
- **FR-10.1** 게시글 관리: 공지글 설정·상단 고정·추천글 지정·메인 노출 여부·숨김·삭제.
- **FR-2.x / FR-5.x 연계**: 공통 게시판(바이브코딩·자동화·수익화·라운지) 전반이 관리 대상.

### 2-2. 모더레이션 / 신고
- **FR-8.1** 글/댓글/자료 신고 (자료 신고 사유: 부적절/악성 파일 의심/저작권/광고·홍보성/파일 안 열림/설명과 다름/기타).
- **FR-8.2** 운영자 **신고 큐** 확인·처리(숨김/삭제), 신고 사유 확인.
- **FR-8.3** **신고 누적 자동 숨김**: 임계치 초과 시 자동 숨김 후 운영자 **검토 큐**로 이동. (임계치 미정 — Open Q-13)
- **FR-8.4** **회원 제재**: 경고 / 일시 정지 / 영구 정지(작성·등록·쪽지 제한). (제재 단계·기준 미정 — Open Q-13)
- **FR-8.5** `[ASSUMPTION]` 금칙어/스팸 필터.
- **FR-10.2** 신고 글/댓글 관리, 댓글 관리.

### 2-3. 실전자료 관리
- **FR-10.3** 실전자료 사후 관리: 목록 관리·신고 자료 확인·숨김·삭제·부적절 첨부 삭제·후기 댓글 관리.
- **FR-4.8** 자료 상태: 공개/임시저장/숨김/삭제됨. 운영자가 숨김/삭제/관리. (검수·공식 배지 미사용)

### 2-4. 회원 관리
- **FR-10.x / FR-8.4 연계**: 제재 대상 회원 조회·상태 관리. (전용 FR 항목은 약함 — 회원 관리 화면 상세는 admin 기획서 의존, Open Q-1)
- **FR-1.7** 회원 탈퇴/계정 삭제 + 작성 콘텐츠 처리 정책(익명화 vs 삭제 미정 — Open Q-12) → 관리자 화면 정책 반영 필요.

### 2-5. SEO 운영
- **FR-10.4** SEO 메타(title/description) 운영자 수정(시스템 자동 생성값 보정).
- **FR-11.1** 메타 자동 생성, 운영자만 보정 가능. **회원에게는 SEO 입력 미노출** → SEO 편집은 관리자 전용 진입점.
- **FR-11.9** `[ASSUMPTION]` 저품질/빈 페이지 noindex 정책 → 관리 판단 개입 여지.

### 2-6. 통계 / 대시보드 / 광고
- **FR-10.5** `[ASSUMPTION]` 광고(애드센스) 관리 영역 + 운영 통계 화면(검색 유입·다운로드 추이 등). 상세는 admin 기획서 기준(Open Q-1).
- 기술 스택 §19가 대시보드 차트 종류를 구체화: 방문자 추이/회원가입 추이/실전자료 다운로드 추이(LineChart), 유입 경로/게시판별 활동/광고 성과(BarChart), 자료 유형 비중(PieChart).
- **게이미피케이션 운영**(FR-9.x): 운영자 **특별 뱃지 수여**(FR-9.3), 포인트 회수/어뷰징 차단은 시스템 자동이나 운영 모니터링 대상.

### 2-7. 관련 NFR
- **NFR-2 (보안)**: 권한 검사 — 관리자 기능은 **클라이언트가 아니라 API 서버에서 권한 검사**(기술 §13, §31-14). 관리자 로그인은 더 강한 rate limiting 적용(§13).
- **NFR-3 (반응형)**: 전 화면 모바일 대응 원칙 — 다만 관리자 화면은 데이터 밀도가 높아 데스크톱 우선 가능성. `[ASSUMPTION]`
- **NFR-5 (접근성)**: 이미지 alt, 명확한 헤딩 구조(전사 원칙, 관리자에도 적용).
- **NFR-6 (백그라운드)**: 통계 집계·이메일·이미지 변환은 BullMQ Worker로 분리 → 관리자 대시보드는 **집계 완료된 캐시 데이터**를 보는 구조(실시간 무거운 쿼리 아님).

## 3. 스테이크 분류 (Stakes Classification)

**분류: internal (내부 운영 도구)**

근거 한 줄: 관리자 사이트는 외부 소비자가 아니라 **소수 운영자만 접근**하는 내부 운영·모더레이션 콘솔이며(공개 노출 없음, 권한 검사로 보호), 실패 시 영향은 사용자 사이트 운영 품질에 간접적 — 따라서 consumer/regulated가 아닌 internal로 분류.
- 단, **개인정보 처리(회원 제재·탈퇴 처리, 한국 개인정보보호법 대응 — FR-14.1)**와 **다운로드/광고 등 수익 지표**를 다루므로, internal 중에서도 "데이터 정확성·권한 통제가 중요한 내부 도구"로 본다.

## 4. 관리자 프론트엔드 기술 스택

> 핵심 원칙: 관리자는 **사용자 사이트와 별도의 디자인/UI를 자체 개발**한다. 외부 관리자 프레임워크·UI 라이브러리를 일절 쓰지 않는다.

- **프레임워크**: 동일 Next.js 16 프로젝트 내 **`/admin` 영역**(`apps/web` 안). React 19 + TypeScript. (기술 §9, §32)
  - 관리자 화면은 SSR보다 **클라이언트 인터랙션 비중 높아도 됨**(SEO 노출 불필요 — 기술 §7 렌더링 원칙).
- **스타일링**: **CSS Modules + 전역 CSS 변수 + Native CSS Nesting**. Tailwind/Sass/styled-components/Emotion **사용 안 함**. (§8)
- **컴포넌트 라이브러리**: **자체 제작 UI 컴포넌트**만. MUI / Ant Design / Chakra / shadcn/ui / **React Admin / Refine / AdminJS / Filament 전부 금지**. (§9, §28, §31-3)
  - 사용자 사이트와 `packages/ui` 컴포넌트를 **일부 공유**하되, 관리자 전용 테이블/필터/폼/차트는 직접 개발. (§9)
- **차트**: **Recharts 3** — 기본 디자인 그대로 쓰지 않고 자체 컴포넌트로 감싸 디자인 시스템 적용(VisitorChart 등 래퍼). (§19)
- **폼/데이터**: React Hook Form + Zod(검증 공유) + TanStack Query(관리자 목록·상태 변경·필터). (§14)
- **인증**: Better Auth + Argon2id. 관리자 로그인은 더 강한 요청 제한. 권한은 **API 서버에서 검사**(클라이언트 권한값만 믿고 관리자 기능 허용 금지 — §13).
- **API**: 관리자 전용 경로 `/api/v1/admin/*`, 별도 권한 검사. (§10)

### 4-1. 별도 디자인 시스템 — 강조점
- 디자인 토큰(`packages/styles`)은 사용자/관리자 **공통 기반**이지만, 관리자는 "데이터 테이블·필터·폼·차트·일괄 작업·확인 모달" 중심의 **별개 UI 패턴군**을 가진다(§9 자체 개발 범위). 즉 토큰은 공유, **레이아웃/컴포넌트 조합과 밀도는 관리자 전용으로 새로 설계**.
- 자체 컴포넌트 후보(§8): Button, Input, Select, Modal, Dialog, Dropdown, Tabs, Badge, Tooltip, Pagination, **Table**, EmptyState, Loading, Toast, FileUploader, **FilterPanel** 등 — 관리자에서 특히 Table/FilterPanel/Pagination/Dialog 비중 큼.

## 5. 접근성 / i18n / 콘텐츠 밀도 / 알림 고려사항

- **i18n**: 단일 언어(**한국어**) 전제. 다국어 요구 **없음**. 한국 개인정보보호법 대응(약관/개인정보처리방침/운영정책 — FR-14.1)이 운영 정책 콘텐츠로 존재. `[ASSUMPTION]` i18n 미적용.
- **접근성**: 전사 NFR-5(이미지 alt, 명확한 헤딩). 관리자 전용 WCAG 등급·키보드 내비 요건은 **명시 없음** → 기본 시맨틱/폼 접근성 수준 가정. `[ASSUMPTION]`
- **콘텐츠 밀도**: 관리자는 본질적으로 **고밀도 데이터 테이블 + 필터 + 일괄 작업** 화면. 신고 큐·회원 목록·자료 목록·통계가 핵심 → 정렬/검색/필터/페이지네이션/벌크 액션이 UX 중심축. (사용자 사이트의 "텍스트형 리스트, 썸네일 배제" 가독성 원칙과 달리, 관리자는 정보 밀도·작업 효율 우선.)
- **알림**: 관리자 대상 알림 채널은 **이메일**(신고 처리 알림·관리자 알림·운영 공지 — 기술 §21)이 명시. 인앱 알림 시스템(FR-12.x)은 주로 일반 회원 대상. 관리자 화면 내 **신고 큐 배지/대기 건수 표시**는 운영 효율상 필요하나 전용 FR 미정 → 설계 시 제안 대상. `[ASSUMPTION]`
- **카운터 지표 모니터링**(PRD 2-3): 스팸/광고성 비율, 신고 처리 대기 건수, 저품질 자료 비율 → 대시보드가 이 "나쁜 신호"를 노출해야 함.

## 6. 관리자 UX가 지켜야 할 제약 (Constraints)

1. **권한 검사는 서버에서** — 화면 권한 분기는 UX 편의일 뿐, 실제 통제는 API(`/api/v1/admin/*`). 클라이언트만 믿고 노출하지 않기. (§13, §31-14)
2. **외부 관리자 UI/템플릿 금지** — React Admin/Refine/AdminJS/Filament/MUI 등 도입 불가. 모든 관리자 UI는 자체 컴포넌트. (§28, §31-3)
3. **관리자 옵션은 회원 화면에 미노출** — 공지/고정/추천/메인노출/SEO 편집은 **별도 관리자 화면에서만**. (FR-10 서두, FR-2.4, §14)
4. **무거운 작업은 비동기** — 통계 집계·이메일·이미지 변환은 Worker로. 관리자 화면은 캐시/완료 데이터 표시(요청 중 SMTP·변환 대기 금지). (§16, §20, §21)
5. **실전자료 파일 보안** — 비공개(`private/resources/`) 저장, 공개 URL 직접 노출 금지, 다운로드는 만료 짧은 서명 URL. 관리자의 "부적절 첨부 삭제"도 이 흐름 위에서. (§15, §31-9)
6. **상태 모델 준수** — 콘텐츠/자료 상태: 공개/임시저장/숨김/삭제됨. 모더레이션 액션이 이 상태 전이를 따라야 함. (FR-4.8)
7. **검수 없음 원칙** — 회원 등록은 즉시 공개. 관리자는 **사전 승인자가 아니라 사후 개입자** → UX는 "검토 대기열 승인"이 아니라 "문제 발생 후 빠른 처리"에 최적화. (FR-4.8, 마스터플랜 §21·§22)
8. **SEO 보정의 신중성** — URL 안정성(NFR-8) 및 noindex 정책은 검색 평판에 직결 → 메타/색인 관련 관리 액션은 되돌리기·미리보기 고려. `[ASSUMPTION]`
9. **모바일 대응 원칙(NFR-3)** vs 데이터 밀도 — 전 화면 모바일 대응이 전사 원칙이나, 관리 테이블은 데스크톱 우선이 현실적. 충돌 해소 필요(아래 Open Q).
