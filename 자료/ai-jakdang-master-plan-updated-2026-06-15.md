# AI작당 전체 기획 최종 정리 업데이트본

이 문서는 AI 커뮤니티 사이트 **AI작당**의 현재까지 확정된 기획을 처음부터 다시 정리한 기준 문서이다.

기존에 작성된 기획 메모, 소스 내 임시 문서, 이전 마크다운 문서가 있더라도 이후 작업에서는 이 문서를 우선 기준으로 삼는다.

특히 이번 업데이트에서는 기존 문서의 내용 중 다음 항목을 현재 논의 기준으로 교체했다.

```text
1. 묻고답하기 상세 구조
2. 답변 채택 방식
3. 실전자료 목록 / 상세 / 등록 구조
4. 실전자료 신뢰도 판단 방식
5. 실전자료 다운로드 중심 정책
6. 실전자료 등록 권한 정책
7. 아직 정해야 할 항목
```

기존 문서와 이번 세션에서 정한 내용이 충돌할 경우, **이번 세션에서 정한 내용을 우선한다.**

---

## 1. 사이트 이름

```text
AI작당
```

### 브랜드 의미

AI작당은 AI를 활용해 무언가를 만들고, 자동화하고, 수익화하려는 사람들이 모이는 실전형 커뮤니티이다.

단순한 AI 뉴스 사이트나 일반 정보 게시판이 아니라, 실제로 AI 도구를 써서 개발, 업무 자동화, 외주, 템플릿 판매, 자동화 구축, 수익화까지 연결하려는 사람들을 위한 공간이다.

---

## 2. 사이트 핵심 콘셉트

```text
AI로 만들고, 자동화하고, 돈으로 연결하는 실전 커뮤니티
```

또는

```text
바이브 코딩, AI 자동화, AI 수익화를 중심으로 한 실전형 AI 커뮤니티
```

---

## 3. 핵심 방향

AI작당은 다음 세 가지를 중심으로 운영한다.

```text
1. 바이브 코딩
2. AI 자동화
3. AI 수익화
```

사용자가 이 사이트에 들어오는 이유는 단순히 AI 정보를 보기 위해서가 아니다.

실제 사용 목적은 다음에 가깝다.

```text
- AI로 개발을 더 빠르게 하기 위해
- AI 코딩 도구를 더 잘 쓰기 위해
- AI 자동화로 업무 시간을 줄이기 위해
- 인력비를 줄이기 위해
- 외주, 템플릿, 자동화 구축 등으로 돈을 벌기 위해
- 막힌 문제나 궁금증을 해결하기 위해
- 실전 자료를 얻기 위해
- 내가 만든 AI 결과물을 공유하고 피드백받기 위해
```

---

## 4. 사이트 차별화 방향

국내에는 이미 AI 관련 커뮤니티와 정보 사이트가 많다.

따라서 AI작당은 단순한 AI 정보 커뮤니티로 가면 차별성이 약하다.

AI작당의 차별점은 다음이다.

```text
AI 도구를 실제로 써먹는 사람들을 위한 실전형 커뮤니티
```

특히 다음 주제에 집중한다.

```text
- 바이브 코딩
- Claude Code, Cursor, Codex 등 AI 코딩 도구 활용
- MCP, Skills, Rules, 프롬프트
- 업무 자동화
- 사업 자동화
- n8n, Make, Zapier, Apps Script 활용
- AI 수익화
- 외주, 템플릿 판매, 자동화 구축 대행
- 실제 제작물 공유
- 실전 질문과 답변
- 다운로드 가능한 실전 자료 공유
```

## 5. 최종 상단 메뉴 구조

최종 상단 메뉴는 다음 6개로 확정한다.

```text
바이브 코딩
AI 자동화
AI 수익화
묻고답하기
실전자료
작당 라운지
```

### 중요 변경 사항

기존에 사용했던 `문제해결`이라는 메뉴명은 폐기한다.

이유는 `문제해결`이라는 이름이 코드 오류, 배포 실패, 자동화 중단처럼 뭔가 실제 작업 중 막힌 문제만 올려야 할 것 같은 좁은 느낌을 주기 때문이다.

AI작당에서는 단순 오류뿐 아니라 다음과 같은 넓은 질문과 소통도 받아야 한다.

```text
- 이 자동화 가능할까요?
- Cursor와 Claude Code 중 뭐가 나을까요?
- 이 외주 견적은 얼마가 적당할까요?
- 이런 AI 서비스 만들면 수익화 가능할까요?
- 프롬프트를 어떻게 짜야 할까요?
- 제가 만든 결과물 방향이 괜찮을까요?
```

따라서 `문제해결` 대신 **묻고답하기**로 변경한다.

## 6. 최종 메뉴 및 하위 메뉴 구조

## 6-1. 바이브 코딩

AI 코딩 도구를 활용해 실제 개발을 하려는 사람들을 위한 공간이다.

```text
바이브 코딩
- 바이브코딩 가이드
- 바이브코딩 팁
```

### 하위 메뉴 역할

```text
바이브코딩 가이드
- 입문, 사용법, 작업 흐름, 운영자가 정리하는 기본 글

바이브코딩 팁
- 짧은 노하우, 프롬프트, 작업 방식, AI 코딩 도구 활용 팁
```

### 삭제한 하위 메뉴

```text
바이브코딩 질문
```

질문은 각 대메뉴 아래에 따로 두지 않고, 모두 `묻고답하기`로 통합한다.

### 들어갈 수 있는 글 예시

```text
- Claude Code로 기존 PHP 프로젝트 수정하는 방법
- Cursor가 프로젝트 구조를 잘못 이해할 때 대응하는 방법
- AI에게 관리자 페이지를 만들게 할 때 요청하는 방식
- AI가 만든 코드를 검수하는 체크리스트
- Codex로 기존 코드베이스를 분석시키는 방법
- Cursor Rules 활용 팁
- Claude Code 작업 흐름 정리
```

## 6-2. AI 자동화

업무 자동화, 사업 자동화, 자동화 도구 활용 사례를 다루는 공간이다.

```text
AI 자동화
- 자동화 가이드
- 자동화 사례
- 자동화 팁
```

### 하위 메뉴 역할

```text
자동화 가이드
- n8n, Make, Zapier, Apps Script 등 자동화 도구 사용법

자동화 사례
- 실제 업무나 사업에 적용한 자동화 흐름 공유

자동화 팁
- 비용 절감, 실패 대응, 도구 선택, 유지보수 노하우
```

### 삭제한 하위 메뉴

```text
자동화 질문
```

질문은 `묻고답하기`로 통합한다.

### 자동화 사례와 자동화 팁의 차이

두 메뉴의 경계가 일부 겹칠 수는 있다.

하지만 커뮤니티에서는 논리적으로 완벽한 분류보다 사용자가 글을 올릴 공간을 명확히 느끼는 것이 중요하다.

```text
자동화 사례
- 실제로 해본 자동화 흐름과 결과 중심

자동화 팁
- 도구 선택, 비용 절감, 오류 예방, 운영 노하우 중심
```

### 들어갈 수 있는 글 예시

```text
- n8n으로 고객 문의 자동 분류하기
- Gmail 문의를 AI가 요약하게 만드는 자동화
- 엑셀 데이터를 AI로 요약하는 방법
- 병원, 학원, 쇼핑몰에서 쓸 수 있는 AI 자동화 사례
- Make와 Zapier 중 어떤 도구가 더 나은가
- 자동화가 실패했을 때 알림을 받는 방법
```

## 6-3. AI 수익화

AI 활용을 돈, 외주, 판매, 비용 절감과 연결하는 공간이다.

```text
AI 수익화
- 외주·판매 팁
- 수익화 사례
```

### 하위 메뉴 역할

```text
외주·판매 팁
- 견적, 제안서, 고객 응대, 템플릿 판매, 자동화 구축 판매 노하우

수익화 사례
- 실제로 돈을 벌었거나 비용을 줄인 경험 공유
```

### 삭제한 하위 메뉴

```text
수익화 질문
```

질문은 `묻고답하기`로 통합한다.

### 제외한 메뉴

```text
수익화 가이드
```

`가이드`라는 이름은 강의나 운영자 정보글 느낌이 강하다.

AI 수익화 영역은 이론보다 실제 경험, 판매 방식, 사례 중심으로 가는 것이 더 낫다.

### 들어갈 수 있는 글 예시

```text
- 바이브 코딩으로 외주 받을 때 견적 잡는 법
- AI 자동화 시스템을 고객에게 제안하는 방법
- 300만 원 이하 AI 개발 외주의 현실
- AI로 만든 템플릿을 판매할 수 있을까
- 자동화 구축 대행을 상품화하는 방법
- AI로 인력비를 줄인 사례
- 프롬프트 자료집이나 전자책 판매 전략
```

## 6-4. 묻고답하기

AI작당의 모든 질문과 답변을 모으는 통합 질문 공간이다.

```text
묻고답하기
- 하위 메뉴 없음
- 카테고리 없음
- 태그로 구분
```

### 메뉴명 결정 이유

`질문답변`은 너무 기능적이고 딱딱하다.

`문제해결`은 너무 좁다.

`묻고답하기`는 질문, 의견, 조언, 방향성 검토, 오류 해결, 수익화 가능성 상담까지 넓게 받을 수 있다.

### 들어갈 수 있는 글 예시

```text
- Cursor 오류가 납니다
- Claude Code를 어떻게 써야 하나요?
- n8n으로 이런 자동화가 가능할까요?
- 이 아이디어 수익화 가능할까요?
- AI 자동화 외주 견적은 얼마가 적당할까요?
- 어떤 AI 툴을 써야 할까요?
- 프롬프트를 어떻게 짜야 할까요?
- 제가 만든 서비스 방향 괜찮을까요?
- 자동화 구축을 고객에게 어떻게 설명해야 할까요?
```

### 묻고답하기 기본 방향

묻고답하기는 네이버 지식인처럼 거창한 답변 게시판으로 만들지 않는다.

질문은 게시글처럼 작성하고, 답변은 댓글형으로 단다.

다만 답변 입력창은 일반 댓글보다 크게 제공하고, 실무 답변에 필요한 기능을 일부 추가한다.

```text
질문 = 게시글형
답변 = 확장 댓글형
```

### 답변 채택 정책

`답변 채택`이라는 강한 기능은 사용하지 않는다.

대신 질문자가 특정 답변에 약하게 표시할 수 있는 기능만 둔다.

```text
도움된 답변
```

또는 UI에서는 다음처럼 표현할 수 있다.

```text
질문자가 도움받은 답변
```

### 도움된 답변 표시 방식

```text
- 질문 작성자만 표시 가능
- 답변 중 1개만 표시 가능
- 언제든 변경 가능
- 포인트, 등급, 보상과 연결하지 않음
- 답변을 마감시키지 않음
- 정답이라는 느낌을 주지 않음
```

이 기능은 정답 채택이 아니라, 질문자가 실제로 참고한 답변을 약하게 알려주는 표시다.

### 상태값

기존의 `답변완료`, `해결완료` 표현은 사용하지 않는다.

묻고답하기 상태값은 다음으로 확정한다.

```text
답변대기
답변있음
해결됨
```

### 상태 기준

```text
답변대기
- 답변이 0개인 질문

답변있음
- 답변이 1개 이상 달린 질문

해결됨
- 질문자가 직접 해결됨으로 변경한 질문
```

### 도움된 답변과 해결됨의 차이

```text
도움된 답변
- 특정 답변에 대한 표시

해결됨
- 질문 전체 상태
```

답변 하나가 도움됐지만 아직 해결되지 않았을 수 있고, 여러 답변을 참고해서 해결했지만 특정 답변 하나를 표시하고 싶지 않을 수도 있다.

따라서 두 기능은 분리한다.

### 묻고답하기 목록 화면

묻고답하기 목록은 공통 게시판보다 상태 표시가 중요하다.

```text
- 게시판명
- 게시판 설명
- 검색창
- 상태 필터
  - 전체
  - 답변대기
  - 답변있음
  - 해결됨
  - 인기질문
- 질문하기 버튼
- 질문 목록
  - 상태 배지
  - 제목
  - 질문 요약
  - 태그
  - 작성자
  - 작성일
  - 조회수
  - 답변 수
  - 좋아요 수
- 페이지네이션
```

### 묻고답하기 목록 예시

```text
[답변대기] Claude Code가 기존 PHP 구조를 계속 잘못 이해합니다
#ClaudeCode #PHP #바이브코딩
작성자 · 2026.06.15 · 조회 82 · 답변 0
```

```text
[해결됨] n8n으로 Gmail 문의를 자동 분류할 수 있을까요?
#n8n #Gmail #자동화
작성자 · 2026.06.15 · 조회 146 · 답변 3
```

```text
- 채택 보상
```

입력 항목이 많으면 질문 작성 진입장벽이 높아진다.

대신 사용자는 자유롭게 글을 쓰고, 필요한 정보는 본문에 직접 작성한다.

### 답변 입력창

답변은 댓글형으로 작성하지만, 일반 댓글보다 조금 더 긴 입력창을 제공한다.

```text
필수 기능
- 여러 줄 텍스트 입력
- 줄바꿈 유지
- 링크 삽입
- 이미지 첨부
- 코드블록
- 등록 버튼

제외 기능
- 제목 입력
- 태그 입력
- 글자 색상
- 형광펜
- H2 / H3
- 파일 첨부
- 복잡한 에디터 기능
```

답변에는 코드, 오류 메시지, 프롬프트, 설정 코드가 포함될 수 있으므로 코드블록은 필요하다.

## 6-5. 실전자료

실전자료는 일반 게시판이 아니다.

AI작당의 실전자료는 기본적으로 **다운로드형 자료실**로 운영한다.

```text
실전자료 = 자료 설명 + 첨부파일 다운로드
```

기존에 논의했던 `복사 중심`, `설치 명령어 직접 입력`, `SKILL.md 직접 입력`, `설정 JSON 직접 입력` 방식은 사용하지 않는다.

자료 자체는 압축 파일이나 문서 파일로 제공하고, 사이트에서는 설명과 다운로드, 후기, 평점 중심으로 보여준다.

### 하위 메뉴

```text
실전자료
- 프롬프트
- MCP·Skills
- Rules·설정
- 템플릿·체크리스트
```

### 자료 유형

등록 화면에서 선택할 자료 유형은 다음으로 둔다.

```text
프롬프트
Claude Code Skill
MCP
Rules·설정
템플릿·체크리스트
```

하위 메뉴명은 `MCP·Skills`로 묶어도 되지만, 등록 자료 유형에서는 `Claude Code Skill`과 `MCP`를 구분한다.

### 하위 메뉴 역할

```text
프롬프트
- 개발 요청 프롬프트, 오류 해결 프롬프트, 자동화 설계 프롬프트

MCP·Skills
- Claude Code Skills, MCP 서버 설정 자료, 자동화 확장 도구 사용 자료

Rules·설정
- Cursor Rules, Claude 설정, 프로젝트 규칙, 코딩 컨벤션

템플릿·체크리스트
- 개발 요청서, 외주 견적서, 자동화 시나리오, 검수 체크리스트
```

### 실전자료의 핵심 방향

```text
- 파일을 올리고
- 설명을 적고
- 다른 사람이 다운로드해서 쓰게 한다
```

실전자료는 사이트 안에서 모든 코드와 설정 내용을 세세하게 관리하지 않는다.

Skill, MCP, Rules, 프롬프트, 템플릿의 실제 내용은 첨부파일 안에 포함한다.

### 허용 파일 형식

```text
.zip
.md
.txt
.json
.pdf
.docx
.xlsx
```

### 자료 유형별 권장 파일 방식

```text
Claude Code Skill
- zip 파일 권장
- SKILL.md, README.md, 예시 파일 포함 가능

MCP
- zip 파일 권장
- 설치 명령어, 설정 JSON, 사용법, 보안 주의사항 포함 권장

Rules·설정
- zip, md, json 파일 권장
- Cursor Rules, CLAUDE.md, settings.json 등을 포함 가능

프롬프트
- md 또는 txt 파일 권장

템플릿·체크리스트
- pdf, docx, xlsx, zip 파일 권장
```

### 실전자료 파일 예시

```text
Claude Code Skill 예시
quality-review-skill.zip

quality-review/
├── SKILL.md
├── README.md
└── examples.md
```

```text
MCP 예시
github-mcp-guide.zip

README.md
install-command.txt
mcp-config.json
usage-example.md
security-note.md
```

```text
Rules·설정 예시
cursor-php-rules.zip

.cursor/
└── rules/
    └── project.mdc
```

```text
프롬프트 예시
code-analysis-prompt.md
prompt.txt
variables.md
example-result.md
```

```text
템플릿·체크리스트 예시
automation-consulting-checklist.xlsx
proposal-template.docx
client-interview-form.pdf
```

## 8. 기존 구조에서 변경된 부분

## 8-1. 문제해결 → 묻고답하기

기존 이름:

```text
문제해결
```

최종 이름:

```text
묻고답하기
```

변경 이유:

```text
- 문제해결은 코드 오류, 배포 실패, 자동화 오류만 올릴 것 같은 느낌이 강하다.
- AI작당에서는 궁금증, 방향 질문, 도구 추천, 수익화 상담, 피드백 요청까지 넓게 받아야 한다.
- 커뮤니티의 소통 느낌을 살리려면 묻고답하기가 더 적합하다.
```

---

## 8-2. 각 대메뉴의 질문 메뉴 삭제

삭제한 메뉴:

```text
바이브코딩 질문
자동화 질문
수익화 질문
```

삭제 이유:

```text
- 질문 게시판이 여러 곳에 흩어지면 글이 분산된다.
- 사용자가 어디에 질문해야 할지 헷갈린다.
- 초기 커뮤니티에서는 질문이 한곳에 모여야 활성화되어 보인다.
```

최종 처리 방식:

```text
모든 질문은 묻고답하기로 통합한다.
각 대메뉴에서는 질문 게시판을 따로 만들지 않는다.
필요하면 각 메뉴 페이지에 [질문하기] 버튼만 배치하고, 클릭 시 묻고답하기 글쓰기 화면으로 이동시킨다.
이때 관련 태그를 자동으로 붙일 수 있다.
```

예시:

```text
바이브 코딩 페이지에서 질문하기 클릭 → 묻고답하기 글쓰기 + 태그: 바이브코딩
AI 자동화 페이지에서 질문하기 클릭 → 묻고답하기 글쓰기 + 태그: AI자동화
AI 수익화 페이지에서 질문하기 클릭 → 묻고답하기 글쓰기 + 태그: AI수익화
```

---

## 8-3. 답변 채택 → 도움된 답변

기존 지식인식 채택 기능은 사용하지 않는다.

최종 방식:

```text
채택 X
정답 표시 X
마감형 답변 X
내공/보상 X

질문자가 도움된 답변 1개를 약하게 표시 가능
```

이유:

```text
- 채택이라는 표현은 부담스럽다.
- 정답을 강제하는 분위기는 자유로운 참여를 막을 수 있다.
- AI작당의 질문은 정답형보다 조언형, 방향 검토형, 경험 공유형이 많다.
```

---

## 8-4. 실전자료 복사형 구조 → 다운로드형 구조

기존에 고려했던 방향:

```text
- 프롬프트 복사
- 설치 명령어 복사
- 설정 JSON 복사
- SKILL.md 복사
- 복사 수 / 사용 수 집계
- 유형별 복잡한 입력폼
```

최종 방향:

```text
- 자료 설명 + 첨부파일 다운로드
- 다운로드 수를 핵심 지표로 사용
- 복사 수 / 사용 수는 사용하지 않음
- 모든 자료 유형은 같은 등록폼 사용
- 실제 자료 내용은 첨부파일 안에 포함
```

변경 이유:

```text
- 개발이 단순해진다.
- Skill, MCP, Rules, 프롬프트, 템플릿을 모두 같은 방식으로 처리할 수 있다.
- 사용자는 파일을 받아서 자기 환경에 적용하는 방식이 더 자연스럽다.
- 복사 수는 사용자에게도 운영자에게도 애매한 지표다.
- 다운로드 수, 평점, 후기 수만으로도 자료 신뢰도를 충분히 판단할 수 있다.
```

## 9. 태그 운영 원칙

AI작당은 게시판을 너무 많이 쪼개지 않는다.

기본 원칙은 다음이다.

```text
카테고리는 적게, 태그는 많이
```

툴별 게시판은 만들지 않는다.

예를 들어 다음 도구들은 독립 게시판이 아니라 태그로 처리한다.

```text
Claude Code
Cursor
Codex
Gemini CLI
n8n
Make
Zapier
MCP
Skills
Rules
PHP
React
Vue
API
배포오류
자동화
수익화
프롬프트
```

### 이유

```text
- 초기 글 수가 부족하면 게시판이 비어 보인다.
- 사용자가 어디에 글을 써야 할지 헷갈린다.
- 관리가 복잡해진다.
- 태그 페이지를 활용하면 나중에 툴별 랜딩 페이지로 확장할 수 있다.
```

### 태그 입력 방식

태그는 자유 입력만 두지 않고, 추천 태그와 자동완성을 제공하는 것이 좋다.

예시:

```text
추천 태그:
Cursor / Claude Code / Codex / n8n / Make / Zapier / MCP / 프롬프트 / 수익화 / 자동화 / PHP / React
```

## 10. 메인 페이지 구성

메인 페이지는 너무 많은 섹션을 노출하지 않는다.

확정된 메인 구성은 다음 6개다.

```text
1. 상단 소개 영역
2. 실전 인기글
3. 묻고답하기 최신글
4. AI 수익화 인기글
5. 실전자료
6. 작당 라운지
```

---

## 10-1. 큰 메인 비주얼은 사용하지 않는다

AI작당은 랜딩페이지가 아니라 정보 탐색형 커뮤니티에 가깝다.

따라서 큰 메인 비주얼 이미지는 사용하지 않는다.

대신 상단에는 짧은 소개 영역만 둔다.

예시 문구:

```text
AI작당
바이브 코딩과 AI 자동화를 실전에서 써먹는 사람들의 커뮤니티
```

또는

```text
AI로 만들고, 자동화하고, 돈으로 연결하는 실전 커뮤니티
```

버튼 예시:

```text
[바이브 코딩 보기] [AI 자동화 보기] [AI 수익화 보기]
```

---

## 10-2. 실전 인기글

바이브 코딩, AI 자동화, AI 수익화 인기글을 하나의 영역으로 묶는다.

탭 예시:

```text
전체 / 바이브 코딩 / AI 자동화 / AI 수익화
```

---

## 10-3. 묻고답하기 최신글

기존의 `문제해결 최신글` 영역은 `묻고답하기 최신글`로 변경한다.

단순 오류 해결뿐 아니라 궁금증, 방향 질문, 수익화 상담, 도구 선택 질문까지 노출될 수 있다.

예시 섹션명:

```text
최근 묻고답하기
```

또는

```text
지금 올라온 질문
```

---

## 10-4. AI 수익화 인기글

AI작당에서 중요한 영역이다.

사용자는 결국 AI를 통해 돈을 벌거나 비용을 줄이고 싶어 한다.

다루는 예시:

```text
- AI 자동화 외주
- 바이브 코딩 외주
- 템플릿 판매
- 자동화 구축 대행
- 인력비 절감 사례
- AI로 만든 서비스 수익화
```

## 12. 공통 게시판 목록 화면

공통 게시판 목록은 **텍스트형 리스트**로 간다.

썸네일은 목록에서 사용하지 않는다.

이미지나 링크 첨부 기능은 제공하되, 목록에서는 썸네일 대신 아이콘만 표시한다.

### 썸네일을 제외하는 이유

이미지가 있는 글과 없는 글이 섞이면 목록이 뒤죽박죽이 되어 보인다.

AI작당은 이미지 커뮤니티가 아니라 글 중심 커뮤니티이므로, 기본 목록은 텍스트형이 가장 안정적이다.

### 목록 화면 구성

```text
- 게시판명
- 게시판 설명
- 검색창
- 정렬 탭
  - 전체
  - 인기글
  - 최신글
  - 댓글 많은 글
- 글쓰기 버튼
- 게시글 목록
  - 제목
  - 요약
  - 태그
  - 작성자
  - 작성일
  - 조회수
  - 댓글 수
  - 좋아요 수
  - 이미지 첨부 아이콘
  - 링크 첨부 아이콘
- 페이지네이션
```

### 목록 예시

```text
[이미지] Claude Code로 관리자 페이지 만들 때 삽질 줄이는 방법
#ClaudeCode #관리자페이지 #바이브코딩
작성자 · 2026.06.15 · 조회 128 · 댓글 4 · 좋아요 12
```

```text

#n8n #자동화 #고객응대
```

```text
- 신고 버튼
```

### 기본 순서

```text
게시판명

작성자 · 작성일 · 조회수 · 댓글수

좋아요 · 공유 · 신고

댓글

수정 / 삭제 / 목록으로
```

### 상세 화면에서 중요한 것

```text
- 에디터로 작성한 내용이 깨지지 않아야 한다.
- 줄바꿈이 유지되어야 한다.
- 이미지가 본문 안에 표시되어야 한다.
- 링크가 자동 인식되거나 링크 카드로 표시될 수 있어야 한다.
- 코드블록이 보존되어야 한다.
- HTML/script는 실행되지 않도록 막아야 한다.
```

### 코드블록 상세 기능

AI작당은 개발, 자동화, 프롬프트, 오류 관련 글이 많으므로 코드블록은 필수다.

코드블록에는 다음 기능이 있으면 좋다.

```text
- 언어 표시
- 복사 버튼
- 가로 스크롤
- 줄바꿈 보존
- 들여쓰기 보존
```

### 외부 링크 표시 방식

초기에는 간단하게 가도 된다.

```text
참고 링크: https://...
```

나중에 고도화하면 링크 카드 형태로 개선할 수 있다.

```text
[링크 카드]
제목
도메인
간단 설명
```

단, OG 이미지 자동 수집은 초기 필수 기능은 아니다.

### 이미지 표시 방식

목록에서는 썸네일을 쓰지 않지만, 상세 페이지에서는 이미지를 본격적으로 보여준다.

```text
- 본문 중간 이미지 삽입 가능
- 첨부 이미지 표시
- 클릭 시 확대 보기
```

초기에는 본문 내 이미지 삽입만 있어도 충분하다.

### 상세 페이지에서 제외하는 것

```text
- 관련글
- 작성자 다른 글
- 북마크
- 별점
- 복잡한 프로필 카드
- 추천 게시글
- 과도한 광고 영역
```

## 15. 본문 에디터 기능

본문 작성에는 에디터를 제공한다.

### 필수 기능

```text
- 굵게
- 제목 H2 / H3
- 목록
- 링크 삽입
- 이미지 삽입
- 코드블록
- 인용
- 폰트 색상
- 형광펜 / 배경 강조
```

### 제외 기능

```text
- 표 삽입
- 복잡한 레이아웃 편집
- 컬럼 나누기
- 글자 크기 자유 조절
- 자유 색상 팔레트
- 과도한 폰트 변경
```

### 표 기능 제외 이유

모바일에서 표는 보기 어렵다.

가로 스크롤이 생기고 글이 좁아져서 커뮤니티 글의 가독성이 떨어진다.

따라서 표 삽입 기능은 초기 에디터에서 제외한다.

비교 정보는 표 대신 목록형으로 작성하도록 유도한다.

예시:

```text
Cursor
- 장점: 기존 코드 수정에 강함
- 단점: 프로젝트가 커지면 컨텍스트 관리 필요

Claude Code
- 장점: 터미널 기반 작업에 강함
- 단점: 초보자는 처음에 어렵게 느낄 수 있음
```

### 폰트 색상 정책

폰트 색상 기능은 넣는다.

다만 완전 자유 색상 팔레트는 제공하지 않는다.

이유는 사용자가 빨강, 파랑, 보라, 형광색을 과도하게 사용하면 글 품질이 낮아 보일 수 있기 때문이다.

권장 방식:

```text
본문
강조
주의
정보
보조
```

또는 제한 색상:

```text
기본색
강조색
빨강
파랑
초록
회색
```

최종 방향:

```text
폰트 색상은 제공하되, 선택 가능한 색상은 제한한다.
형광펜 / 배경 강조 기능도 제공한다.
```

## 16. 코드블록 정책

AI작당은 코드, 에러 메시지, 프롬프트, 설정 파일, 자동화 JSON 등을 다루는 글이 많다.

따라서 코드블록 기능은 필수다.

### 필요한 이유

일반 본문에 코드를 붙여 넣으면 다음 문제가 생긴다.

```text
- 줄바꿈이 깨짐
- 들여쓰기가 사라짐
- 특수문자가 변형됨
- HTML 태그가 실행될 위험이 있음
- 에러 메시지를 정확히 읽기 어려움
```

### 구현 방향

별도의 코드 작성 폼을 만들기보다는, 글쓰기 에디터 안에 `코드블록 삽입` 기능을 넣는다.

사용자는 글을 쓰다가 필요한 순간에 코드블록을 삽입하면 된다.

### 코드블록 필수 기능

```text
- 줄바꿈 보존
- 들여쓰기 보존
- 특수문자 보존
- HTML/script 실행 방지
- 복사 버튼
- 가로 스크롤
```

### 있으면 좋은 기능

```text
- 언어 선택
- 문법 색상 강조
```

```text
첨부파일
```


예시:

```text
이 자료는 Claude Code에서 GitHub MCP를 연결할 때 필요한 설정 파일과 사용 예시를 묶은 자료입니다.
```

```text

2. 압축을 풀고 README.md를 확인합니다.
3. install-command.txt 또는 mcp-config.json 내용을 본인 환경에 맞게 적용합니다.
4. Claude Code 또는 Cursor에서 연결 상태를 확인합니다.
```

MCP, Skill, Rules에는 주의사항이 중요하다.

```text
- 외부 서비스 계정 인증이 필요할 수 있습니다.
- API Key가 공개 저장소에 노출되지 않도록 주의해야 합니다.
- 신뢰할 수 없는 파일은 실행하지 마세요.
- 다운로드한 자료는 본인 환경에 맞게 확인 후 적용해야 합니다.
```

### 후기 댓글 영역

별도 오류 제보 기능은 만들지 않는다.

### 신고하기

문제 있는 자료는 별도 오류 제보가 아니라 신고하기로 처리한다.

신고 사유:


모바일에서는 하단 고정 버튼을 둘 수 있다.


## 21. 실전자료 등록 화면

실전자료 등록 화면은 다운로드형 자료실 기준으로 단순화한다.

자료 유형별 복잡한 입력폼은 만들지 않는다.

모든 자료는 같은 등록폼을 사용한다.

### 등록 권한

```text
비회원
- 등록 불가
- 목록/상세 열람 가능

일반 회원
- 실전자료 등록 가능
- 본인이 등록한 자료 수정/삭제 가능

운영자
- 모든 자료 수정/숨김/삭제 가능
- 신고된 자료 관리 가능
```

실전자료는 커뮤니티 참여를 위해 누구든 등록 가능해야 한다.

운영자만 등록하는 방식은 사용하지 않는다.

### 등록 흐름

```text
자료 유형 선택
↓
공통 정보 입력
↓
첨부파일 등록
↓
사용 방법 / 주의사항 작성
↓
태그 입력
↓
미리보기
↓
등록
```

### 기본 입력 항목

```text
자료명
한 줄 설명
자료 유형
지원 환경
난이도
자료 설명
첨부파일
사용 방법
주의사항
관련 링크
버전
태그
```

### 필수 입력

```text
자료명
한 줄 설명
자료 유형
지원 환경
첨부파일
자료 설명
태그
```

### 선택 입력

```text
난이도
사용 방법
주의사항
관련 링크
버전
```

단, MCP, Skill, Rules 자료는 주의사항 입력을 강하게 권장한다.

필수로 강제하지는 않는다.

### 자료 유형 선택

```text
프롬프트
Claude Code Skill
MCP
Rules·설정
템플릿·체크리스트
```

자료 유형을 선택해도 복잡한 추가 입력칸이 열리지 않는다.

대신 유형별 안내 문구만 다르게 보여준다.

### 유형별 안내 문구 예시

```text
프롬프트
- 프롬프트 본문을 md 또는 txt 파일로 정리해서 올려주세요.
```

```text
Claude Code Skill
- SKILL.md, README.md, 예시 파일을 zip으로 묶어 올리는 것을 권장합니다.
```

```text
MCP
- 설치 방법, 설정 JSON, 사용 예시, 주의사항을 zip 안에 포함하는 것을 권장합니다.
```

```text
Rules·설정
- Cursor Rules, CLAUDE.md, settings.json 등 적용 파일을 zip 또는 md/json 파일로 올려주세요.
```

```text
템플릿·체크리스트
- docx, xlsx, pdf, zip 형식으로 올리는 것을 권장합니다.
```

### 첨부파일 정책

허용 파일:

```text
.zip
.md
.txt
.json
.pdf
.docx
.xlsx
```

권장 방식:

```text
Skill / MCP / Rules
- zip 권장

프롬프트
- md, txt 권장

템플릿·체크리스트
- pdf, docx, xlsx, zip 권장
```

파일 개수:

```text
첨부파일 최대 3개
대표 다운로드 파일 1개 지정 가능
```

목록과 상세에서는 대표 파일 기준으로 다운로드 수를 집계한다.

### 등록 화면 버튼

```text
[미리보기]
[임시저장]
[등록하기]
[취소]
```

### 등록 후 상태

검수는 따로 두지 않는다.

상태는 단순하게 간다.

```text
공개
임시저장
숨김
삭제됨
```

회원이 등록하면 바로 공개된다.

다만 신고가 많거나 문제가 있는 자료는 운영자가 숨김 처리할 수 있다.

### 등록 화면에서 삭제할 것

기존 검토안에서 아래 항목은 제외한다.

```text
- SKILL.md 내용 직접 입력
- 설치 명령어 직접 입력
- 설정 JSON 직접 입력
- 프롬프트 본문 직접 입력
- Rules 설정 코드 직접 입력
- 유형별 복잡한 전용 입력칸
- 복사 버튼용 별도 코드 블록
- 복사 수 / 사용 수 집계
- 운영자 검수 상태
- 검수됨으로 등록
- 공식 자료로 등록
```

## 22. 실전자료 평가 / 신뢰도 정책

실전자료는 운영자 검수 없이 누구나 등록 가능하게 한다.

자료의 신뢰도는 사용자 반응으로 판단한다.

### 신뢰도 지표

```text
평점
다운로드 수
후기 수
좋아요 수
조회 수
```

### 목록에 표시할 지표

목록에는 너무 많은 지표를 보여주지 않는다.

```text
★ 4.7 · 다운로드 1,240 · 후기 32
```

이 정도가 가장 깔끔하다.

### 상세에 표시할 지표

```text
- 평점
- 다운로드 수
- 좋아요 수
- 후기 수
- 조회 수
```

### 평점 방식

```text
1점 ~ 5점
```

평점은 사용자가 실제로 자료를 사용해본 뒤 남기는 것이 좋다.

다만 초기에는 엄격한 사용 검증을 하지 않는다.

### 후기 방식

후기는 일반 댓글형으로 간다.

```text
- 사용 후기
- 설치 경험
- 적용 팁
- 최신 버전에서의 작동 여부
- 다른 환경에서의 사용 경험
```

### 좋아요

좋아요는 가볍게 추천하는 기능으로 둔다.

좋아요 수는 상세에서 보여주고, 목록에서는 필요하면 후순위로 보여준다.

### 다운로드 수

다운로드 수는 대표 다운로드 파일 기준으로 자동 집계한다.

복사 수, 사용 수는 집계하지 않는다.

### 오류 제보 기능

별도의 `오류 제보` 기능은 만들지 않는다.

문제 제기는 다음 방식으로 처리한다.

```text
- 후기 댓글
- 신고하기
```

일반 커뮤니티 자료실에서는 오류 제보를 별도 기능으로 두는 경우가 많지 않다.

초기에는 기능을 단순하게 유지한다.

### 신고하기

신고 사유는 다음으로 둔다.

```text
- 부적절한 자료
- 악성 파일 의심
- 저작권 문제
- 광고 / 홍보성 자료
- 파일이 열리지 않음
- 설명과 다른 자료
- 기타
```

운영자는 신고된 자료를 확인해 숨김 또는 삭제 처리할 수 있다.

### 사용하지 않는 배지 / 지표

```text
- 검수됨
- 공식
- 회원제보
- 복사 수
- 사용 수
- 오류 제보 수
```

검수 시스템을 운영하지 않기 때문에 `검수됨`, `공식` 배지는 사용하지 않는다.

## 23. 관리자 기능 방향

회원 글쓰기 화면에는 관리자용 옵션을 노출하지 않는다.

관리자만 사용할 수 있는 기능은 별도 관리자 화면에서 처리한다.

### 관리자용 게시글 기능

```text
- 공지글 설정
- 상단 고정
- 추천글 지정
- 메인 노출 여부
- 숨김 처리
- 삭제 처리
- 신고 글 관리
- 댓글 관리
```

### 실전자료 관리자 기능

실전자료는 누구나 등록 가능하므로 관리자 기능은 검수가 아니라 사후 관리 중심이다.

```text
- 실전자료 목록 관리
- 신고된 자료 확인
- 자료 숨김 처리
- 자료 삭제 처리
- 부적절한 첨부파일 삭제
- 후기 댓글 관리
- 신고 사유 확인
```

### 회원 글쓰기 화면에서 제외

```text
- 공지글 설정
- 상단 고정
- 추천글 지정
- 메인 노출 여부
- 숨김 처리
- 검수 상태 설정
- 공식 자료 설정
```

이런 기능이 회원 화면에 보이면 글쓰기 화면이 복잡해진다.

## 24. SEO와 AI 검색 대응 방향

AI작당은 검색엔진과 AI 검색이 내용을 이해하기 쉽게 구조화해야 한다.

### 기본 원칙

```text
- 페이지마다 고유한 title 생성
- 페이지마다 meta description 생성
- canonical URL 설정
- H1은 페이지당 하나만 사용
- H2, H3 구조 명확히 사용
- breadcrumb 구조 사용
- 카테고리, 태그 페이지 최적화
- 게시글 요약문 자동 생성 또는 별도 저장
- 이미지 alt 작성
- 첨부파일 설명 작성
- sitemap.xml 자동 생성
- robots.txt 설정
- OG 태그 설정
- JSON-LD 구조화 데이터 출력
```

### 게시글 유형별 구조화 데이터 방향

```text
운영자 정보글: Article 또는 BlogPosting
회원 커뮤니티 글: DiscussionForumPosting
묻고답하기 글: QAPage
실전자료 상세: SoftwareSourceCode, CreativeWork, DigitalDocument 중 적합한 타입 검토
프로필 페이지: ProfilePage
카테고리 목록: CollectionPage
사이트 전체: WebSite, Organization
경로 표시: BreadcrumbList
```

실전자료의 구조화 데이터는 자료 유형에 따라 달라질 수 있으므로 개발 단계에서 최종 확정한다.

### 태그 페이지 활용

툴별 게시판을 만들지 않는 대신, 태그 페이지를 SEO 랜딩 페이지로 활용한다.

예시:

```text
/tags/cursor
/tags/claude-code
/tags/n8n
/tags/mcp
/tags/ai-automation
```

## 26. 초기 운영 전략

초기에는 회원들이 알아서 글을 써주기를 기대하면 안 된다.

운영자가 직접 핵심 글을 먼저 쌓아야 한다.

다만 실전자료는 누구나 등록 가능하게 열어둔다.

### 초기 운영 방식

```text
1. 운영자가 직접 핵심 글을 작성한다.
2. 검색 유입 가능한 실전 글을 먼저 쌓는다.
3. Claude Code, Cursor, MCP, Skills 관련 자료를 정리한다.
4. AI 자동화 사례를 운영자가 직접 만든다.
5. 묻고답하기에는 자주 나올 질문을 미리 만들어 둔다.
6. 실전자료에는 다운로드 가능한 샘플 자료를 먼저 올린다.
7. 태그별 랜딩 페이지를 강화한다.
8. 검색 유입이 생긴 뒤 커뮤니티 기능을 강화한다.
9. 애드센스는 콘텐츠가 충분히 쌓인 뒤 신청한다.
```

## 27. 수익화 방향

초기 수익화는 애드센스를 고려할 수 있다.

하지만 현실적으로 애드센스만으로 큰 수익을 기대하기는 어렵다.

AI작당에서 가장 먼저 기대할 수 있는 수익은 **외주 문의 유입**일 가능성이 높다.

### 수익 모델 후보

```text

2. 외주 문의 유입

4. 프롬프트 자료집 판매
5. AI 자동화 기획서 판매

7. 유료 강의
8. 유료 커뮤니티
9. AI 자동화 구축 대행
10. 템플릿 마켓
```

### 현실적 우선순위

```text

```

### 실전자료와 수익화 연결

초기 실전자료는 무료 다운로드 중심으로 간다.

나중에 자료 품질과 트래픽이 쌓이면 다음으로 확장할 수 있다.

```text

- 유료 체크리스트
- 프롬프트 자료집
- 자동화 기획서 양식
- Claude Code / Cursor 세팅 패키지
```

하지만 MVP에서는 유료 결제와 자료 마켓은 넣지 않는다.

## 32. 현재까지 최종 결론

AI작당은 다음 방향으로 간다.

```text
바이브 코딩, AI 자동화, AI 수익화를 중심으로 한 실전형 AI 커뮤니티
```

최종 상단 메뉴는 다음이다.

```text
바이브 코딩
AI 자동화
AI 수익화
묻고답하기
실전자료
작당 라운지
```

질문은 각 대메뉴에 흩어두지 않는다.

```text
모든 질문은 묻고답하기로 통합한다.
```

묻고답하기는 지식인식 답변 게시판이 아니라 확장 댓글형으로 간다.

```text
질문은 게시글처럼 작성한다.
답변은 댓글형으로 단다.
답변 채택은 사용하지 않는다.
질문자가 도움된 답변을 약하게 표시할 수 있게 한다.
상태값은 답변대기 / 답변있음 / 해결됨으로 둔다.
```

공통 게시판은 텍스트형 리스트를 기본으로 한다.

```text
목록에는 썸네일을 쓰지 않는다.
이미지와 링크는 아이콘으로만 표시한다.
상세 페이지에서 이미지와 링크를 본격적으로 보여준다.
```

글쓰기 화면은 단순하게 간다.

```text
제목
본문 에디터
태그
이미지 첨부
외부 링크
파일 첨부
임시저장
등록 / 취소
```

에디터는 실전 글 작성에 필요한 기능만 제공한다.

```text
굵게
제목 H2 / H3
목록
링크
이미지
코드블록
인용
폰트 색상
형광펜 / 배경 강조
```

표 기능은 제외한다.

실전자료는 다운로드형 자료실로 간다.

```text
자료 설명
첨부파일 다운로드
평점
다운로드 수
후기 댓글
신고하기
```

실전자료는 누구든 등록 가능하다.

```text
운영자만 등록하는 구조로 가지 않는다.
검수 시스템은 두지 않는다.
자료의 신뢰도는 평점, 다운로드 수, 후기 수로 판단한다.
오류 제보 전용 기능은 두지 않고 후기 댓글과 신고하기로 처리한다.
```

AI작당의 핵심은 예쁜 기능이 아니라, 사용자가 실제로 AI를 활용해 만들고, 자동화하고, 수익화하는 과정에서 필요한 정보를 찾고 소통할 수 있게 만드는 것이다.
