# 요구사항 브리프 — 팔로우 & 유저 프로필

> 상태: 기획 추가 → **epics.md 반영 완료(2026-06-22)**. 아래 내용은 `_bmad-output/planning-artifacts/epics.md`에 FR-1.10·FR-7.10 및 Story 1.2/1.3/1.9/1.10 수정 + 신규 Story 5.12로 통합됨.
> 작성일: 2026-06-22
> 후속: PRD 본문(`_bmad-output/planning-artifacts/prds/...`)에 FR-1.10·FR-7.10 트레이스 추가는 메인 에이전트 검토 권장(이번 작업 범위는 epics/stories).

## 1. 배경 / 목적
커뮤니티 성격 강화를 위해 **유저 간 팔로우**와 **공개 유저 프로필 페이지**를 추가한다.
- 유저 계정(작성자명 등)을 클릭하면 해당 유저의 공개 프로필로 이동
- 프로필에서 그 유저가 작성한 글 목록을 모아 볼 수 있음
- 각 유저는 자기 프로필을 일부 꾸밀 수 있음
데이터가 쌓이기 전 초기 단계에 소셜 그래프를 넣어 마이그레이션 비용을 줄인다.

## 2. 핵심 개념 분리 (중요)
세 개념을 한 페이지로 합치지 않는다.
1. **마이페이지 `/mypage`** (기존): 내가 보는 나 — 비공개 포함(북마크/좋아요 등). 탭 구조 유지.
2. **공개 프로필 `/users/[nickname]`** (신규): 남이 보는 나 — 작성 글, 소개, 팔로워/팔로잉 수, 등급 뱃지, 팔로우/DM 버튼. **반드시 별도 라우트.** 비공개 데이터 노출 금지.
3. **팔로우 그래프**: 팔로우/언팔, 카운트.

## 3. 닉네임 정책 (확정)
- 닉네임은 **유니크**. → 별도 `handle` 컬럼 불필요, `nickname` 하나로 표시·프로필 URL·@멘션을 모두 처리.
- DB에 `nickname` **unique 제약** 추가 (현재 `users.nickname`은 `notNull`만 있고 unique 아님 → 변경).
- **가입 즉시 자동 배정**: 시스템이 유니크한 닉네임 + 기본 프로필 이미지를 자동 제공. 사용자는 이후 프로필 수정에서 변경.
- URL 안전성: 허용 문자 정책 필요(한글·영문·숫자·`_` 권장), 길이 제한. 닉네임 변경 시 unique 재검증.

### 3.1 자동 닉네임 생성
- 방식 추천: `형용사 + 명사 + 숫자` 한국어 조합 (예: `명랑한너구리482`) — 서비스 톤(잭당)에 맞고 충돌률 낮음.
- 충돌 처리: insert 시 unique 위반 → 숫자만 바꿔 최대 N회 재시도 → fallback(더 긴 숫자). **트랜잭션 내 처리로 동시가입 레이스 방지.**

### 3.2 기본 프로필 이미지
- 원본: `자료/프로필/` 내 PNG 11종 (장당 ~2MB).
- **반드시 최적화 후 사용**: 256×256(또는 512) WebP로 리사이즈 → `apps/web/public/` 또는 오브젝트 스토리지에 배치. 코드는 경로만 참조.
- 가입 시 11종 중 랜덤 1개 배정. 균등 분배 권장(카운터 기반 또는 crypto 랜덤; `Math.random` 지양).
- 저장 방식: 기본 이미지는 `defaultAvatarIndex`(인덱스)로 저장하면 "미변경=기본" 구분 및 추후 이미지 교체에 유리. 커스텀 업로드 시 `avatarUrl`로 덮어씀.

## 4. 프로필 꾸미기 범위 (확정: 약하게)
MVP는 **약하게**로 시작:
- 아바타 이미지(커스텀 업로드 가능, 기본은 랜덤 배정)
- 배너 이미지
- 한 줄 소개(bio)
- 외부 링크
- ※ 자유 레이아웃/위젯/사용자 HTML(싸이월드식)은 **제외**(공수·XSS·모더레이션·디자인 시스템 일관성 리스크). 추후 별도 판단.

## 5. 데이터 모델 영향 (개략)
- `users` 확장: `nickname` unique 제약, `avatarUrl`, `defaultAvatarIndex`, `bio`, `bannerUrl`.
- 신규 `follows`: (`follower_id`, `following_id`, `created_at`) — 복합 PK, 자기 자신 팔로우 방지, 양방향 인덱스. 카운트는 집계 쿼리 or 캐시 컬럼.
- ORM: Drizzle (`packages/database/src/schema/`). 테이블·컬럼 snake_case, 코드 타입 camelCase.

## 6. 기존 자산 연계
- 등급 뱃지: `lib/ranks` + `RankBadge` — 프로필에 표기(경로 하드코딩 금지).
- 알림 `/notifications`: 팔로우 발생 시 알림 발송 연계.
- 메시지 `/messages`: 프로필에 DM 버튼 연계.

## 7. 단계 제안 (MVP → 확장)
1. `nickname` unique + 자동 닉네임 생성 + 기본 프로필 이미지 랜덤 배정 (가입 플로우 수정)
2. `/users/[nickname]` 공개 프로필 + 작성 글 목록 + 작성자명 클릭 → 프로필 이동
3. `follows` 테이블 + 팔로우/언팔 버튼 + 팔로워·팔로잉 카운트 + `/mypage` 팔로잉/팔로워 탭
4. 프로필 꾸미기(아바타/배너/소개/링크) 편집 UI

## 8. 후속(이번 범위 제외)
- **팔로잉 피드**(내가 팔로우한 사람들의 최신 글 모아보기): 라운지 변형으로 별도 기능. 추후 판단.

## 9. 미결정 / 메인 에이전트가 정할 것
- 자동 닉네임 단어 풀(형용사/명사 사전) 구체화
- 닉네임 변경 빈도 제한(예: 30일 1회) 여부
- 프로필 비공개/차단 정책
- 아바타/배너 업로드 스토리지(로컬 public vs S3 등) 및 용량·포맷 제한

# 요구사항 브리프 — AI 창작물 스펙 & 게시판 추가 (2026-06-22)

> 상태: **PRD + epics.md 반영 완료**. FR-5.2·FR-5.3 신규, FR-5.1 갱신, epics Story 2.1 board 시드 갱신 + 신규 Story 2.11(창작 스펙)·2.12(작당 의뢰소).

## A. AI 창작물 스펙 (FR-5.2 — AI 창작마당 전용, 선택)
사용자 요청: AI로 만든 영상/이미지를 올릴 때 사용 AI 툴·프롬프트 등 "스펙"을 등록해 창작물 사양을 알게 함. 스펙은 글 작성 시 **선택**. 상세 페이지 **우측**에 스펙 표시.
- 항목(작성자 보강, 전부 선택): 창작물 유형(이미지/영상/오디오·음악/3D·기타), 사용 AI 툴·모델(다중·파이프라인), 프롬프트(positive)·네거티브, 주요 파라미터(시드·화면비/해상도·길이/fps·steps/CFG/sampler 등 자유 key-value), 후처리/워크플로(업스케일·인페인팅·편집툴·ControlNet/LoRA·img2img 참조), 비용(유료/무료)·제작 소요 시간, 라이선스·상업적 사용 가능 여부
- 데이터: `post_creative_spec`(1:1 post). 프롬프트/네거티브는 코드블록형·복사 버튼. 데스크톱 우측 패널 / 모바일 본문 하단 접힘.

## B. 작당 수다방 (FR-5.1 — "자유 게시판" 리네이밍)
- 이름: **작당 수다방**(절대 "자유게시판" 명칭 안 씀, 사용자 요청). URL `/lounge/free` → `/lounge/talk`.
- 주제 제한 없는 AI 잡담·수다. 공통 게시판 구조 그대로.

## C. 작당 의뢰소 (FR-5.3 — 구인·외주 게시판 신규)
- 이름: **작당 의뢰소**(작성자 결정). URL `/lounge/gigs`. 외주 받고 서로 소통하는 공간.
- 글 작성 형식(작성자 설계): 글 유형(의뢰/구직, 필수)·분야(다중, 필수)·모집 상태(모집중/마감, 필수)·연락 방법(쪽지 기본+외부 선택, 필수)·예산/기간/진행 방식(선택)·본문.
- 상세: 의뢰 정보 카드 + [쪽지 보내기](Epic 7 연계) + 모집중/마감 배지. 거래 보증 없음·사기 주의 고지 필수. 신고/차단 연계.

## D. 미결정 / 메인 에이전트가 정할 것
- 창작 스펙 파라미터 표준 키 셋(툴별 상이) — 자유 key-value로 시작, 추후 정규화 검토
- 작당 의뢰소 분야 카테고리 최종 목록 + 마감 글 자동 처리(기간 경과 시 자동 마감?) 정책
- 의뢰소 거래 분쟁/신고 운영 정책, 외부 연락처 노출 범위(스팸·개인정보)
- 두 게시판 이름(작당 수다방/작당 의뢰소) 최종 확정 — 작성자 제안값, 변경 시 알려주면 일괄 리네이밍
